-> Issue 1: SQL Riesgo de inserción (Seguridad)
  └─Severidad: Blocker (Crítica)  
  └─Archivo/Línea: backend/routes/subject.js L26  

**Descripción:**  
El código construye una consulta SQL directamente con datos controlados por el usuario, lo que lo hace vulnerable a ataques de inyección SQL. Por ejemplo dejando la libertad de hacer "name": "users; DROP TABLE subjects; --"

**Recomendación:**  
Se recomienda cambiar la forma en que se interactua con SQL haciendo que se revise los parametros de la consulta SQL para asi separar la logica de datos y el codigo SQL.

**Acción:**  
Se modificara la consulta SQL revisando antes si el parametro {name} contiene solo caracteres, numeros o guiones bajo, esto se realizara para la proxima actualizacon del backend.

--------------------------------------------------------------------

-> Issue 2: Ineficiencia en la verificación de existencia (Mantenibilidad)
  └─Severidad: Minor (media)
  └─Archivo/Línea: backend/index.js L12

**Descripción:**  
SonarCloud detectó que la constante ALLOWED_EMAILS se define como un arreglo (Array) que se utiliza únicamente para verificar si un correo existe o no dentro de la lista. Esto es ineficiente, ya que el método .includes() recorre todo el arreglo en cada verificación (complejidad O(n))

**Recomendación:**  
Usar un objeto Set, que está optimizado para búsquedas rápidas con el método .has(), el cual tiene complejidad O(1). Esto mejora el rendimiento del código.

**Acción:**  
Se considera recoregir el codigo modificando el metodo .includes() por .has(), esto ya que no afecta la funcionalidad del codigo.

---

## Conclusión y acciones de mejora
Finalmente ambas recomendaciones ehcas por SonarCloud se implementaran debido a que son  utilies para nuestro proyecto y no muy complejas de modificar.
- Las recomendaciones **críticas (Seguridad)** se abordarán de inmediato.  
- Las recomendaciones **menores (Mantenibilidad)** se aplicarán durante el mantenimiento del código.
